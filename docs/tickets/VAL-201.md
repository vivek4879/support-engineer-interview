# VAL-201: Email validation accepted special-case issues

## Priority and Impact
- Priority: High
- Impact: Users can be confused when the system silently changes email casing, and typo domains (for example `.con`) cause account/login issues later.

## Root Cause
1. Backend signup schema used `z.string().email().toLowerCase()`, which silently transformed user input.
2. Backend had no guard for common typo domains such as `.con`.
3. User lookup in auth flows used exact-case match, which can lead to case-sensitivity friction.
4. Frontend email pattern check was basic and did not cover special-case guidance.

## Fix Implemented
1. Added shared email utility in `lib/utils/email.ts`:
   - `normalizeEmailForLookup(...)` for case-insensitive lookup normalization.
   - `validateEmailForSignup(...)` for special-case checks.
   - `getCommonEmailTypoSuggestion(...)` now uses `mailcheck` for typo-domain suggestions.
2. Updated backend signup validation in `server/routers/auth.ts`:
   - Removed silent lowercase transform from Zod schema.
   - Added email checks via `superRefine` with clear user-facing messages.
3. Updated backend user lookup in `server/routers/auth.ts`:
   - Signup conflict and login queries now use `lower(users.email)` comparison for case-insensitive lookup.
4. Added DB-level protection in `lib/db/index.ts`:
   - `CREATE UNIQUE INDEX IF NOT EXISTS users_email_lower_unique ON users(lower(email));`
5. Updated frontend signup form in `app/signup/page.tsx`:
   - Uses stronger email pattern and shared special-case validation for immediate feedback.
6. Added regression tests in `tests/email.test.ts`.
7. Added local TypeScript module typing for `mailcheck` in `mailcheck.d.ts`.

## Why This Resolves the Issue
- System no longer silently alters what the user typed.
- Common typo domains are blocked early with a correction hint.
- Email matching is case-insensitive for auth lookups, reducing login/signup friction.
- DB unique index on lowercase email prevents case-variant duplicate accounts.

## Preventive Measures
1. Keep normalization logic separate from displayed/stored user-entered value when silent mutation is not desired.
2. Enforce canonical uniqueness rules at DB level, not only in application logic.
3. Add validation tests for user-input edge cases before shipping form or auth changes.

## Production Note
For this challenge, I implemented typo suggestion with `mailcheck` and stopped at signup-time validation plus case-insensitive lookup/uniqueness.
In production, I would additionally implement email verification and deliverability/risk checks (for example provider-backed verification flow, bounce handling, and reputation checks).
I am not implementing those here because they require external provider integration, operational setup, and infrastructure beyond this repository's challenge scope/timebox.

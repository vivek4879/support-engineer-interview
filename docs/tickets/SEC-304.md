# SEC-304: Session management allowed multiple valid sessions per user

## Priority and Impact
- Priority: High
- Impact: Old sessions remained valid after new logins, increasing risk of unauthorized access from previously issued tokens.

## Root Cause
`signup` and `login` each inserted a new row in `sessions` without invalidating existing sessions for that user.

Result: one user could have multiple simultaneous valid session tokens in the database.

## Fix Implemented
1. Enforced single-session policy:
   - Added `replaceUserSession` helper in `lib/utils/session.ts`.
   - On signup/login, we now delete existing sessions for the user and insert the new one in a DB transaction.
2. Standardized session expiry/cookie max-age constants in the same helper (`SESSION_MAX_AGE_SECONDS`, `createSessionExpiryIso`).
3. Hardened logout invalidation:
   - Logout now parses session token directly from request cookies and deletes by token even if `ctx.user` is null.
4. Unified cookie parsing:
   - Added `lib/utils/cookies.ts` and reused it in both `server/trpc.ts` and `server/routers/auth.ts`.

## Why This Resolves the Issue
- New login/signup rotates the session, so only one active session remains for that user.
- Stale/previous session tokens are invalidated immediately.
- Logout now reliably clears session state based on cookie token, reducing invalidation gaps.

## Preventive Measures
1. Define explicit session policy (single-session vs multi-device) and enforce consistently in auth flows.
2. Centralize session lifecycle helpers (token extraction, rotation, expiry) to avoid drift.
3. Add regression tests for session rotation and cookie token parsing.

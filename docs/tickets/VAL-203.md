# VAL-203: State code validation accepted invalid values (for example, `XX`)

## Priority and Impact
- Priority: Medium
- Impact: Invalid state codes can break address verification and banking communications.

## Root Cause
Validation only checked format (`2 letters`) instead of checking against real USPS state/territory codes.
- Frontend accepted any two letters using `/^[A-Z]{2}$/`.
- Backend schema accepted any two letters via `z.string().length(2).toUpperCase()`.

## Fix Implemented
1. Added shared state validation utility in `lib/utils/state.ts`:
   - `normalizeUsStateCode(...)`
   - `isValidUsStateCode(...)`
   - Supports USPS codes for US states, DC, and territories.
2. Updated backend signup validation in `server/routers/auth.ts`:
   - Added `superRefine` check that rejects unknown codes.
3. Updated frontend signup form in `app/signup/page.tsx`:
   - Keeps 2-letter format check.
   - Adds shared `isValidUsStateCode` validation for immediate user feedback.
4. Added regression tests in `tests/state.test.ts`.

## Why This Resolves the Issue
- `XX` and other unknown codes are now rejected both in UI and API.
- Real valid codes (for example `CA`, `NY`, `PR`) are accepted consistently.
- Shared utility prevents frontend/backend validation drift.

## Preventive Measures
1. Use canonical code lists (like USPS codes) instead of format-only regex for enumerated fields.
2. Keep validation logic in shared utilities used by both client and server.
3. Add regression tests for known invalid and valid boundary values.

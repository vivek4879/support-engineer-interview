# VAL-205: Zero amount funding requests were accepted

## Priority and Impact
- Priority: High
- Impact: Zero-dollar funding requests create unnecessary transaction attempts and noisy records.

## Root Cause
Validation gap existed across layers:
1. Frontend amount input used `min: 0.0` on a text field, which allowed `0.00` to pass client checks.
2. Server validation did not explicitly enforce a post-normalization minimum, so effective-zero values needed stricter guarding.

## Fix Implemented
1. Added shared amount guard in `lib/utils/money.ts`:
   - `isValidFundingAmount(amount)` requires effective amount >= $0.01.
2. Enforced server-side funding minimum in `server/routers/account.ts`:
   - Zod input amount now refines with `isValidFundingAmount`.
   - Added explicit runtime guard after normalization (`Amount must be at least $0.01`).
3. Updated frontend amount validation in `components/FundingModal.tsx`:
   - Replaced weak `min: 0.0` with explicit validation rules:
     - minimum `$0.01`
     - maximum `$10,000`
4. Added regression tests in `tests/money.test.ts`.

## Why This Resolves the Issue
- Backend blocks funding amounts below $0.01 even if client-side validation is bypassed.
- Frontend now prevents obvious zero-value submissions before API call.
- Shared helper ensures consistent threshold logic.

## Preventive Measures
1. Avoid numeric `min/max` rules on text inputs without explicit parse/validate checks.
2. Enforce monetary thresholds on backend after normalization.
3. Keep amount-policy logic centralized and covered by regression tests.

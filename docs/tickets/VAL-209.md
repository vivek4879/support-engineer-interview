# VAL-209: Amount input accepted values with multiple leading zeros

## Priority and Impact
- Priority: Medium
- Impact: Input like `00010.50` creates confusing display and user trust issues during funding.

## Root Cause
Funding amount input validation only checked numeric shape and min/max thresholds.
It did not enforce canonical format rules for the whole-number part, so multiple leading zeros passed (`00010`, `01.25`).

## Fix Implemented
1. Added shared amount-input validator in `lib/utils/money.ts`:
   - `validateFundingAmountInput(...)` now enforces:
     - required input
     - valid numeric format with up to 2 decimals
     - no multiple leading zeros in whole part
     - existing min/max policy checks
2. Updated funding form validation in `components/FundingModal.tsx`:
   - Replaced fragmented regex/min/max checks with single shared validator.
   - Keeps amount parsing explicit at submit using trimmed input.
3. Added regression tests in `tests/money.test.ts` for leading-zero rejection and valid canonical inputs.

## Why This Resolves the Issue
- Inputs like `00010` and `01.25` are now rejected before submission.
- Valid inputs like `10`, `10.50`, and `0.01` are still accepted.
- Validation logic is centralized, reducing drift and inconsistent error behavior.

## Preventive Measures
1. For user-entered currency strings, validate canonical format separately from numeric value checks.
2. Centralize input-policy checks in shared utility functions rather than duplicating form rules.
3. Add regression tests for formatting edge cases (leading zeros, decimal precision, empty values).

## Scope Note
The API receives amount as a numeric value; leading-zero semantics are a text-input concern at UI entry time.
So this fix is intentionally at the input-validation layer where the raw string is available.

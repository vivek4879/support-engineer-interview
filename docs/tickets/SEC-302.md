# SEC-302: Account numbers generated with insecure randomness

## Priority and Impact
- Priority: High
- Impact: Predictable account-number generation increases risk of enumeration and abuse.

## Root Cause
`server/routers/account.ts` generated account numbers using:
- `Math.random()`

`Math.random()` is not cryptographically secure and can be predictable in attack scenarios.

## Fix Implemented
1. Added secure account-number helper in `lib/utils/account-number.ts`:
   - Uses `crypto.randomInt(...)` for cryptographically stronger randomness.
   - Enforces strict range/integer checks.
2. Updated account creation flow in `server/routers/account.ts` to use `generateSecureAccountNumber()` instead of `Math.random()`.
3. Added regression tests for formatting and range guardrails.

## Why This Resolves the Issue
- `crypto.randomInt` is designed for security-sensitive randomness.
- Account numbers are now much harder to predict compared to `Math.random()`.
- Validation in helper prevents malformed/out-of-range generator output from being silently accepted.

## Preventive Measures
1. Don't use `Math.random()` in security-sensitive identifiers.
2. Centralize identifier generation logic in audited utility modules.

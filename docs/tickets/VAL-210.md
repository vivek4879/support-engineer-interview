# VAL-210: Card type detection missed valid cards

## Priority and Impact
- Priority: High
- Impact: Valid cards from some networks could be rejected due to incomplete card-type handling.

## Root Cause
Card validation relied on generic format + checksum checks without explicit card-network detection coverage.
This left gaps in card-type recognition logic and made support behavior less explicit for multiple valid network ranges.

## Fix Implemented
1. Added explicit card-type detection in `lib/utils/card.ts`:
   - `detectCardType(...)` now recognizes:
     - Visa
     - Mastercard (including 2-series range)
     - American Express
     - Discover
     - JCB
     - Diners Club
     - UnionPay
2. Updated `isValidCardNumber(...)` to require:
   - recognized supported card type
   - plus existing checksum and sanity checks
3. Expanded card tests in `tests/card.test.ts` to cover multiple valid network examples and unsupported-type rejection.

## Why This Resolves the Issue
- The system now has explicit, broader card-type recognition logic rather than implicit coverage.
- Valid cards from more networks are accepted consistently.
- Unsupported/unknown card prefixes are rejected deterministically.

## Preventive Measures
1. Keep card-network patterns centralized in one utility module.
2. Add test cases for each supported network and known range expansions.
3. Review network ranges periodically as issuer standards evolve.

## Production Note
For this challenge scope, I expanded in-app card-type detection and validation coverage.
In production, the preferred approach is using hosted gateway components (for example Stripe Elements or Adyen Web Drop-in) to offload card validation and network-rule updates, and to avoid direct handling of raw card data in our application.

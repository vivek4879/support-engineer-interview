# SEC-301: SSNs stored in plaintext

## Priority and Impact
- Priority: Critical
- Impact: Plaintext SSNs in DB are a severe privacy/compliance risk in case of database exposure.

## Root Cause
During signup, user input (including `ssn`) was inserted directly into the `users` table:
- `await db.insert(users).values({ ...input, password: hashedPassword })`

Because `input.ssn` was raw digits, SSN values were stored in plaintext.

## Fix Implemented
1. Added SSN hashing utility (`lib/utils/ssn.ts`) using `bcrypt`.
2. Updated signup flow in `server/routers/auth.ts` to hash SSN before insert.
3. Updated auth responses (signup/login) to omit both `password` and `ssn` from returned user payloads.

## Why This Resolves the Issue
- SSN values are no longer persisted as readable plaintext.
- Even if DB data is exposed, SSNs are not directly visible.
- API responses no longer leak SSN fields back to clients.

## Preventive Measures
1. Treat SSN as high-sensitivity data with dedicated handling helpers.
2. Ban direct spread of raw registration payloads into persistence for sensitive fields.
3. Add security code review checks for at-rest protection and response redaction.
4. In production, consider key-managed encryption/tokenization and migration of existing plaintext records.

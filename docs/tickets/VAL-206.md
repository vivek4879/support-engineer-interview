# VAL-206: Invalid card numbers were accepted

## Priority and Impact
- Priority: Critical
- Impact: Invalid payment-card data could pass validation, causing avoidable funding failures and customer frustration.

## Root Cause
Card validation was weak and inconsistent:
1. Backend (`server/routers/account.ts`) accepted any `fundingSource.accountNumber` string for `type: "card"` (no Luhn or format validation).
2. Frontend only checked basic pattern/prefix logic, which still allowed many invalid card numbers.

## Fix Implemented
1. Added shared card-validation utility in `lib/utils/card.ts`:
   - Normalizes spaces/hyphens
   - Validates digit length range
   - Rejects repeated-digit placeholders
   - Runs Luhn checksum validation
2. Enforced backend validation in `server/routers/account.ts` using Zod `superRefine`:
   - For card funding, reject request when `isValidCardNumber(...)` is false.
3. Aligned frontend validation in `components/FundingModal.tsx`:
   - Uses same shared `isValidCardNumber(...)`
   - Normalizes card number before API submission.
4. Added tests in `tests/card.test.ts`.

## Why This Resolves the Issue
- Backend now blocks invalid card numbers even if client-side checks are bypassed.
- Frontend gives earlier feedback with the same validation rules, reducing failed submissions.
- Shared utility keeps validation logic consistent across layers.

## Preventive Measures
1. Always enforce payment validation on server-side (client checks are advisory only).
2. Use shared validation utilities to prevent frontend/backend drift.
3. Add regression tests for known valid/invalid test card numbers.

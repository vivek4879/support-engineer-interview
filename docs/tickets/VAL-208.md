# VAL-208: Weak password requirements

## Priority and Impact
- Priority: Critical
- Impact: Weak passwords increase risk of credential stuffing and account compromise.

## Root Cause
Backend signup validation only enforced `password: z.string().min(8)`, which allows many weak passwords.

Frontend had extra checks, but backend is the security boundary; client-side rules can be bypassed.

## Fix Implemented
1. Added shared password policy utility (`lib/utils/password.ts`) with requirements:
   - Minimum length: 12
   - Not in common-password denylist
   - Must include lowercase, uppercase, number, and special character
2. Updated backend signup schema in `server/routers/auth.ts`:
   - Uses `superRefine` + `validatePasswordStrength(...)`
   - Rejects weak passwords server-side with clear message
3. Updated frontend signup form in `app/signup/page.tsx`:
   - Uses same shared validator for consistent UX
   - Updated minimum length message to 12 characters
4. Added tests in `tests/password.test.ts`.

## Why This Resolves the Issue
- Server now blocks weak passwords regardless of client behavior.
- Frontend guidance matches backend policy to reduce user confusion.
- Shared validator prevents policy drift between layers.

## Preventive Measures
1. Treat password policy as security logic owned by backend.
2. Keep one shared validator used by both backend and UI.
3. Add regression tests for each policy rule and failure message.

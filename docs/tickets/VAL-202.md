# VAL-202: Date of birth validation accepted invalid/future DOBs

## Priority and Impact
- Priority: Critical
- Impact: Allowing future or underage DOB values creates regulatory/compliance risk for account onboarding.

## Root Cause
Signup validation only required `dateOfBirth` to be present:
- Frontend: required field only
- Backend: `dateOfBirth: z.string()` with no DOB semantics

As a result, invalid/future dates and underage DOBs could pass.

## Fix Implemented
1. Added shared DOB validator in `lib/utils/date-of-birth.ts`:
   - Enforces `YYYY-MM-DD` format
   - Rejects impossible dates
   - Rejects future DOBs
   - Enforces minimum age 18
2. Updated backend signup validation in `server/routers/auth.ts`:
   - Added DOB check in schema `superRefine`
3. Updated frontend signup form in `app/signup/page.tsx`:
   - Added DOB validator in field registration for immediate UX feedback
4. Added regression tests in `tests/date-of-birth.test.ts`.

## Why This Resolves the Issue
- Server now blocks non-compliant DOBs even if client checks are bypassed.
- Frontend and backend use the same DOB policy, preventing validation drift.
- Future and underage DOB entries are rejected deterministically.

## Preventive Measures
1. Keep compliance-sensitive rules in shared backend-first validators.
2. Avoid treating date fields as plain strings without semantic validation.
3. Add boundary tests around age eligibility and date validity.

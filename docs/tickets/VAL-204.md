# VAL-204: International phone number validation accepted invalid numeric strings

## Priority and Impact
- Priority: Medium
- Impact: Bad phone numbers reduce ability to contact customers for important notifications.

## Root Cause
Validation was inconsistent and format-only:
1. Frontend required exactly 10 digits, which blocked many valid international formats.
2. Backend accepted a broad 10-15 digit numeric pattern, which allowed generic digit strings without proper international structure.
3. No shared validator existed, so frontend/backend rules diverged.

## Fix Implemented
1. Added shared phone utility in `lib/utils/phone.ts`:
   - `normalizePhoneNumber(...)` strips separators and converts `00` prefix to `+`.
   - `isValidInternationalPhoneNumber(...)` enforces E.164-like format (`+` followed by 8-15 digits, non-zero country-code start).
   - `validateInternationalPhoneNumber(...)` provides a clear error message.
2. Updated backend signup validation in `server/routers/auth.ts`:
   - Replaced regex-only phone check with shared validator in `superRefine`.
   - Normalizes phone before insert to store consistent values.
3. Updated frontend signup form in `app/signup/page.tsx`:
   - Uses shared phone validator for immediate feedback.
   - Updated placeholder to international example format.
4. Added regression tests in `tests/phone.test.ts`.

## Why This Resolves the Issue
- Random numeric strings like `1234567890` are now rejected.
- Valid international numbers (for example `+14155552671`) are accepted.
- Same validation logic runs on both frontend and backend, so behavior is consistent.
- Stored phone values are normalized, reducing downstream formatting issues.

## Preventive Measures
1. Use shared validation utilities for fields validated in both UI and API.
2. Prefer standardized formats for contact fields (international/E.164-like representation).
3. Cover normalization + validation behavior with regression tests.
